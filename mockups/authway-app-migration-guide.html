Phân loại app — Nhóm A / B / C / Shared

Xác định nhóm trước khi migrate (effort khác nhau rất nhiều)

Nhóm A

IP whitelist / VPN only

Đặc điểm: App không có concept "user". Ai vào được mạng/VPN thì xài. Không có login form, không có DB users, không có session.

Ví dụ: Dashboard Grafana nội bộ chưa setup auth, internal docs/wiki, admin panel chạy trong subnet, Streamlit dashboard read-only.

Effort: 0 LOC code · ~15 phút infra · Low risk
Chỉ thêm oauth2-proxy ở edge — app code không biết gì cả. App tự nhiên có per-user audit ở Zitadel level, dù app không hiểu user là ai.
Nhóm B

Account tự tạo (login form + password)

Đặc điểm: App có DB users với password_hash, login form, register, reset password, session cookie, data ownership per-user (vd posts.user_id).

Ví dụ: CMS, project management, ticket system, app CRUD có "my data" view.

Effort: 50–150 LOC code · 2–4h · Medium risk (DB migration)
Bỏ toàn bộ auth code, thay bằng middleware đọc header. Schema migration: password_hash ra, zitadel_subject vào. Auto-provision user lần đầu login.
Nhóm C

Không support middleware / framework hạn chế

Đặc điểm: Stack không có middleware pattern truyền thống, không inject được auth check trước mọi request. Hoặc framework quá cũ/đặc thù để thay đổi.

Ví dụ: Streamlit < 1.37, Gradio, Jupyter Notebook deploy, Shiny R, app PHP legacy không có middleware hook.

Effort: 10–30 phút · 2 strategy
Strategy 1: Treat như Nhóm A (oauth2-proxy + IP whitelist, app không biết user). Mất per-user audit ở app level nhưng Zitadel vẫn có.
Strategy 2: Upgrade lên version có support (Streamlit ≥1.37 có st.context.headers) → trở thành Nhóm B.
Shared

1 account dùng chung cả team

Đặc điểm: App có login form nhưng cả team dùng chung 1 account (admin/Admin@2024). Không có per-user data, không có audit.

Ví dụ: Admin panel infra (server-mgmt, DNS, mail) — pattern phổ biến nhất gây risk.

Effort: ~10 LOC (bỏ login form) · 30 phút · Low risk
Bỏ login form, mỗi member có Zitadel user riêng → MFA + audit per-user + revoke centralized. Đây là case mang lại GIÁ TRỊ AN NINH lớn nhất (chính là lý do dùng Zitadel ban đầu).

1Provision VPS + Docker + UFW
curl -fsSL https://get.docker.com | sh
sudo apt install docker-compose-plugin gettext-base
sudo ufw allow from <VPN_SUBNET> to any port 22,80,443
sudo ufw default deny incoming
sudo timedatectl set-ntp true   # NTP quan trọng cho TOTP
NTP phải sync — TOTP code phụ thuộc đồng hồ. Lệch >30s = login fail. UFW chỉ allow VPN subnet — defense in depth, đề phòng port leak.
2Generate secrets + render config
openssl rand -base64 32   # ZITADEL_MASTERKEY → lưu Bitwarden + sealed envelope
openssl rand -hex 24      # POSTGRES_PASSWORD
openssl rand -hex 24      # ZITADEL_DB_PASSWORD
Mất ZITADEL_MASTERKEY = backup vô dụng. Mọi TOTP seed, OIDC client_secret, refresh token trong DB được mã hoá bởi key này. Backup ≥2 nơi.

▸ Next.js (App Router) 
// app/page.tsx
import { headers } from 'next/headers';
const h = headers();
const email = h.get('x-auth-request-email');
const sub = h.get('x-auth-request-user');
// ... user đã được auth, không cần check thêm
▸ Express / Fastify 
app.use((req, res, next) => {
  const email = req.get('X-Auth-Request-Email');
  if (!email) return res.status(401).end();
  req.user = {
    email,
    sub: req.get('X-Auth-Request-User'),
    name: req.get('X-Auth-Request-Preferred-Username'),
  };
  next();
});
▸ FastAPI 
from fastapi import Header, HTTPException, Depends

def get_user(x_auth_request_email: str = Header(None)):
    if not x_auth_request_email:
        raise HTTPException(401)
    return {"email": x_auth_request_email}

@app.get("/me")
def me(user=Depends(get_user)):
    return user
▸ Django 
# settings.py
MIDDLEWARE = [

    email = request.headers.get('X-Auth-Request-Email')

Header được oauth2-proxy inject TRƯỚC khi forward. App tin tưởng header này vì traffic không thể bypass oauth2-proxy (network segmentation + strip-auth-in middleware ở Traefik xoá header attacker gửi vào).
6DB schema migration (chỉ nhóm B)
-- Backup trước
ALTER TABLE users RENAME TO users_legacy;   -- giữ forensic 30 ngày

CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  zitadel_subject TEXT UNIQUE NOT NULL,     -- immutable, bind theo sub
  name TEXT,
  created_at TIMESTAMP DEFAULT now()
);

Auto-provision khi user login lần đầu:

middleware:
  email = X-Auth-Request-Email
  sub = X-Auth-Request-User
  user = SELECT * FROM users WHERE zitadel_subject = sub
  if not user:
      user = INSERT (email, zitadel_subject, name) RETURNING *
  request.user = user
Bind theo zitadel_subject (UUID immutable) thay vì email. Nếu user đổi email trong Zitadel sau này → vẫn match cùng user. Email-based bind = risk tạo user trùng khi đổi email.

3 · Vì sao chọn IAP pattern (oauth2-proxy) thay vì OIDC trong app code?
Tiêu chíOIDC trong app code (truyền thống)IAP với oauth2-proxy (chọn)
App dev phải hiểu OIDC? Có — phải biết flow code/PKCE, token refresh, state, nonce Không — chỉ đọc HTTP header
LOC change per app 200–500 LOC + library mới 15–50 LOC, 0 library
Implementation bảo mật 10 app = 10 implementation = 10 cơ hội sai 1 implementation chuẩn (oauth2-proxy) dùng chung
Update OIDC library Per app, có thể vỡ Update 1 chỗ (oauth2-proxy image)
Cross-language support Cần library cho mọi stack (Streamlit không có) Header HTTP — universal, mọi stack
Tách concern Auth + business logic trộn lẫn Auth ở edge, app chỉ lo business logic
Migrate khỏi Zitadel sau này Phải sửa code mọi app Đổi config oauth2-proxy, app không sửa
Trade-off: Phải maintain thêm component oauth2-proxy + Traefik trên mỗi app-vps. Nhưng cost này 1-time setup, đổi lại app dev không phải đụng OIDC. Với 4–10 app, ROI rõ ràng.

Universal

Prompt 1 — Scan + Đề xuất plan (chạy TRƯỚC)

Dùng khi chưa biết app thuộc nhóm nào. AI phân tích codebase và recommend approach.

Tôi cần migrate app này sang dùng centralized auth qua Zitadel (pattern IAP — Identity-Aware Proxy với oauth2-proxy sidecar).

Cách hoạt động:
- oauth2-proxy đặt trước app, handle toàn bộ OIDC flow với Zitadel
- App KHÔNG cài OIDC library, KHÔNG implement callback
- App chỉ đọc HTTP header `X-Auth-Request-Email`, `X-Auth-Request-User` (UUID sub),
  `X-Auth-Request-Preferred-Username` để biết user là ai
- Logout = link `/oauth2/sign_out`

Reference đầy đủ: https://github.com/<org>/authway/blob/master/mockups/authway-app-migration-guide.html

Yêu cầu:
1. Scan codebase này, xác định app thuộc nhóm nào:
   - Nhóm A: IP whitelist only, không có user concept → 0 LOC code change
   - Nhóm B: có login form + DB users + password_hash → cần bỏ auth + DB migration
   - Nhóm C: framework không support middleware (Streamlit cũ, Gradio…) → strategy riêng
   - Shared: 1 account cả team dùng chung → bỏ login form
2. Liệt kê chính xác các file cần SỬA, XÓA, hoặc THÊM (paths + lý do)
3. Liệt kê tất cả thư viện/dependency cần GỠ (passport, next-auth, authlib, devise…)
4. Nếu nhóm B: viết SQL migration cho schema (rename users → users_legacy, thêm zitadel_subject UNIQUE)
5. Đề xuất middleware/dependency snippet phù hợp stack hiện tại
6. KHÔNG implement gì cả ở bước này — chỉ trả lời plan dạng checklist

Trả lời ngắn gọn, dùng tiếng Việt cho phần giải thích, code/path bằng tiếng Anh.
Nhóm A / Shared

Prompt 2A — Bỏ login form, đọc header (≤30 phút)

App này thuộc Nhóm A/Shared. Migrate sang IAP với oauth2-proxy:

1. Bỏ login form, register, reset password (xóa file routes/components liên quan)
2. Bỏ session cookie tự quản, bỏ dependency auth (liệt kê + gỡ khỏi package.json/requirements.txt)
3. Bỏ middleware auth cũ (nếu có), thay bằng middleware đọc header:
   - `X-Auth-Request-Email` (bắt buộc, nếu missing → 401)
   - `X-Auth-Request-User` (UUID sub, immutable, dùng làm primary key user)
   - `X-Auth-Request-Preferred-Username` (display name)
4. Update logout link/button → `Logout`
5. Update UI nếu cần show user info: lấy từ header, KHÔNG fetch /api/me phức tạp
6. Update env vars: bỏ AUTH_SECRET, JWT_SECRET, OAUTH_*, GOOGLE_CLIENT_ID… không còn dùng

KHÔNG được:
- Cài thư viện OIDC (passport, next-auth, authlib, openidconnect…)
- Implement OAuth callback handler /oauth2/callback (oauth2-proxy đã làm)
- Verify JWT signature
- Quản lý refresh token
- Đọc header từ client request (chỉ đọc khi container chạy sau Traefik+oauth2-proxy)

Sau khi sửa: chạy build/test để verify không break. Nếu không có test, viết 1 test giả lập header `X-Auth-Request-Email: test@example.com` → route protected trả 200.
Nhóm B

Prompt 2B — Full migration (2–4h, có DB)

App này thuộc Nhóm B (có DB users với password_hash). Migrate sang IAP với oauth2-proxy:

== Code changes ==
1. Bỏ TẤT CẢ:
   - Login / register / reset password routes + UI form + email verify flow
   - Password hashing logic (bcrypt/argon2/scrypt verify + hash)
   - Session cookie tự quản (express-session, iron-session…)
   - Dependency: passport, next-auth, devise, django-allauth, authlib, jsonwebtoken… liệt kê hết
2. Thêm middleware đọc header (~15-30 LOC):
   - Header `X-Auth-Request-Email` (required → 401 nếu missing)
   - Header `X-Auth-Request-User` (UUID sub, lưu vào DB)
   - Header `X-Auth-Request-Preferred-Username` (display name)
3. Auto-provision: nếu user với `zitadel_subject` chưa tồn tại trong DB → INSERT row mới
4. Update mọi reference `req.user.id` / `current_user.id` cũ → vẫn dùng internal id từ DB lookup
5. Logout: link ``

== DB migration ==
Viết migration file cho framework hiện tại (Prisma/TypeORM/Alembic/Django migrations…):

```sql
-- Backup forensic: rename thay vì DROP
ALTER TABLE users RENAME TO users_legacy;

CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  zitadel_subject TEXT UNIQUE NOT NULL,  -- immutable UUID từ Zitadel sub
  name TEXT,
  created_at TIMESTAMP DEFAULT now(),
  updated_at TIMESTAMP DEFAULT now()
);

-- Foreign key tables (posts, orders, etc.) cần chỉnh:
-- Option 1: chấp nhận data loss (TRUNCATE CASCADE) — đã được approve
-- Option 2: bảo lưu, mapping thủ công sau khi user re-login
```

QUAN TRỌNG: KHÔNG bind theo email vì user có thể đổi email trong Zitadel sau này. Bind theo `zitadel_subject` (UUID immutable).

== Verify ==
- Build/test xong không lỗi compile
- Test giả lập: set 3 header `X-Auth-Request-*` → request protected route → 200 + user được auto-provision trong DB nếu chưa có
- Test logout: GET /oauth2/sign_out → status 302 redirect

Trả lời từng bước, hiển thị diff hoặc full file mới khi sửa.
Tuỳ stack

Prompt 3 — Thêm context stack-specific (gắn vào Prompt 2)

Thêm 1 trong các đoạn dưới đây vào CUỐI Prompt 2 tuỳ stack:

▸ Next.js (App Router) 
Stack: Next.js App Router, TypeScript.
- Dùng `headers()` từ `next/headers` trong Server Components
- Middleware `middleware.ts` ở root: redirect / return 401 nếu thiếu header
- Server Actions / API routes (`route.ts`): `const email = headers().get('x-auth-request-email')`
- Client Component muốn dùng user info: pass từ Server Component xuống qua props,
  hoặc fetch `/api/me` (route.ts đọc header rồi return JSON)
- Bỏ NextAuth: gỡ `next-auth`, xóa `[...nextauth]/route.ts`, gỡ `SessionProvider`
▸ Express / Fastify 
Stack: Express (hoặc Fastify), Node.js.
- Bỏ `passport`, `express-session`, `cookie-parser` (nếu chỉ dùng cho auth)
- 1 middleware ~15 LOC trước mọi route protected:
  app.use((req, res, next) => {
    const email = req.get('X-Auth-Request-Email');
    if (!email) return res.status(401).end();
    req.user = { email, sub: req.get('X-Auth-Request-User') };
    next();
  });
- Update tất cả handler dùng `req.user`
▸ FastAPI 
Stack: FastAPI, Python.
- Bỏ `python-jose`, `passlib`, `authlib`, `fastapi-users` (nếu có)
- Dependency:
  from fastapi import Header, HTTPException, Depends
  def get_user(x_auth_request_email: str = Header(None),
               x_auth_request_user: str = Header(None)):
      if not x_auth_request_email: raise HTTPException(401)
      return {"email": x_auth_request_email, "sub": x_auth_request_user}
- Update mọi route: `def route(user=Depends(get_user)):`
▸ Django 
Stack: Django.
- Dùng built-in `RemoteUserMiddleware` + `RemoteUserBackend`
- Custom middleware nhỏ map `HTTP_X_AUTH_REQUEST_EMAIL` → `request.META['REMOTE_USER']`
- settings.py:
  MIDDLEWARE = [..., 'myapp.middleware.ZitadelHeaderMiddleware',
                     'django.contrib.auth.middleware.RemoteUserMiddleware']
  AUTHENTICATION_BACKENDS = ['django.contrib.auth.backends.RemoteUserBackend']
- Gỡ `django-allauth`, `django-rest-auth`, `djangorestframework-simplejwt`
- User model: dùng built-in `User`, lưu `zitadel_subject` qua `User.username` hoặc Profile model
▸ Flask 
Stack: Flask.
- Gỡ `Flask-Login`, `Flask-Security`, `Authlib`
- Before-request handler:
  @app.before_request
  def authenticate():
      email = request.headers.get('X-Auth-Request-Email')
      if not email: abort(401)
      g.user_email = email
      g.user_sub = request.headers.get('X-Auth-Request-User')
- Update mọi nơi dùng `current_user` → `g.user_email`
▸ Streamlit ≥ 1.37 
Stack: Streamlit (kiểm tra version với `streamlit --version`, phải ≥1.37).
- Đầu mọi file page:
  import streamlit as st
  email = st.context.headers.get('X-Auth-Request-Email')
  if not email:
      st.error("Auth required"); st.stop()
  st.session_state['user_email'] = email
- Logout button: `st.link_button("Logout", "/oauth2/sign_out")`
- Nếu Streamlit < 1.37 → upgrade trước, hoặc fallback Nhóm A (oauth2-proxy + IP whitelist, app không biết user)
Infra

Prompt 4 — Tạo docker-compose + Traefik + oauth2-proxy

Sau khi app code đã migrate, chạy prompt này để AI tạo infra. Cần điền 4 thông tin từ Zitadel admin console.

Tạo file `docker-compose.yml` + `traefik.yml` + `.env.example` cho app này deploy lên VPS riêng, theo pattern IAP với oauth2-proxy + Traefik.

Thông tin app:
- App hostname: {{app.company.com}}
- App image / Dockerfile: {{ghcr.io/team/myapp:latest hoặc ./Dockerfile}}
- App listening port (trong container): {{3000}}
- Zitadel issuer URL: {{https://auth.company.com}}
- Zitadel hostname: {{auth.company.com}}
- Zitadel VPS IP (để extra_hosts nếu lab self-signed): {{192.168.x.x}}

Yêu cầu:
1. `docker-compose.yml` với 3 service:
   - traefik (v3.4) — port 80/443, TLS, route Host(`{{app.company.com}}`)
   - oauth2-proxy (v7.7.1) — provider=oidc, client từ env, set-xauthrequest=true,
     pass-access-token=false, cookie-refresh=1h, upstream=app container
   - app — KHÔNG có traefik labels (chỉ reachable qua oauth2-proxy)
2. `.env.example` với:
   - APP_HOSTNAME, ZITADEL_HOSTNAME, ZITADEL_ISSUER_URL, ZITADEL_IP
   - APP_CLIENT_ID (lấy từ Zitadel console), APP_CLIENT_SECRET, APP_COOKIE_SECRET (32 chars)
3. `traefik.yml` static config: entrypoints, providers docker
4. README.md ngắn ghi 5 bước deploy:
   - Tạo OIDC app trên Zitadel console, copy client_id + secret
   - Fill .env
   - Generate cookie secret: `openssl rand -base64 32 | tr -d '/+=' | cut -c1-32`
   - Thêm `192.168.x.x auth.company.com` vào /etc/hosts (nếu lab)
   - `docker compose up -d`

Reference template chuẩn: https://github.com/<org>/authway/blob/master/infra/app-vps/docker-compose.yml

KHÔNG dùng pass-access-token=true (leak token xuống app). Dùng `--ssl-insecure-skip-verify=true` CHỈ nếu Zitadel cert là self-signed (lab); production thì bỏ flag này.
Workflow đề xuất cho member:
  1. Mở Claude Code / Cursor / Continue trong workspace app
  2. Paste Prompt 1 → AI scan, classify nhóm, đưa plan
  3. Đọc plan, confirm hoặc điều chỉnh
  4. Paste Prompt 2A hoặc 2B + đoạn Prompt 3 tương ứng stack → AI sửa code
  5. Review diff, commit branch migrate/zitadel-iap
  6. Paste Prompt 4 để AI tạo infra files
  7. Tạo OIDC app trên Zitadel console, fill .env, deploy
  8. Test e2e + report lại auth admin
Kiểm tra sau khi AI sửa: human review BẮT BUỘC, không merge blind. Đặc biệt check: (1) không còn route public chấp nhận password, (2) không còn endpoint nào skip middleware, (3) DB migration không drop data nhầm, (4) không hardcode header trust khi test local mà quên gỡ.
