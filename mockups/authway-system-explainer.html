What is Authway

Một "auth brain" duy nhất cho 5 devs — login 1 lần, vào mọi internal tool, có MFA + audit log.

Authway tập trung identity của team vào Zitadel self-host (1 VPS). Mỗi app vibe-code chạy trên VPS riêng có oauth2-proxy sidecar. Dev/admin gõ URL app → bị redirect tới auth.authway.lab → login với local user + MFA Passkey → callback set cookie SSO → app load với header X-Auth-Request-Email nói "Hello {email}". Defense in depth: IP whitelist subnet 192.168.122.0/24 + VPN vẫn giữ, Zitadel chỉ là layer identity.

3  Login pipeline — IAP redirect flow

sequenceDiagram autonumber participant U as Browser (dev) participant TA as traefik (app-vps) participant OP as oauth2-proxy participant TZ as traefik (auth-server) participant Z as zitadel participant A as app downstream U->>TA: GET https://app.authway.lab/ TA->>TA: strip-auth-in (xoá X-Auth-* từ client) TA->>OP: forwardAuth /oauth2/auth OP-->>TA: 401 (no cookie) TA-->>U: 302 /oauth2/start?rd=... U->>OP: GET /oauth2/start OP-->>U: 302 https://auth.authway.lab/authorize?client_id=... U->>TZ: GET /authorize TZ->>Z: forward Z-->>U: login form U->>Z: username + password Z-->>U: MFA prompt (Passkey/TOTP) U->>Z: WebAuthn assertion / OTP Z-->>U: 302 callback?code=... U->>OP: GET /oauth2/callback?code=... OP->>TZ: token exchange (code → access/id token) TZ->>Z: forward Z-->>OP: tokens OP->>OP: validate, set cookie (.authway.lab) OP-->>U: 302 / (original) U->>TA: GET / (with cookie) TA->>OP: forwardAuth /oauth2/auth OP-->>TA: 202 + headers X-Auth-Request-Email, ... TA->>A: forward + headers A-->>U: HTML "Hello user@authway.lab"
Highlights
  • • Strip trước khi forwardAuth: middleware strip-auth-in CHẠY TRƯỚC iap-auth để xoá header X-Auth-* mà client gửi vào → chống spoof.
  • • Cookie scope .authway.lab: leading dot → SSO giữa các subdomain (app A login xong, app B vào thẳng).
  • • MFA bắt buộc trong Zitadel, không phải oauth2-proxy. Policy enforced server-side → không bypass được.
  • • Token chỉ ở oauth2-proxy, app không thấy access_token (set --pass-access-token=false) — giảm blast radius nếu app bị XSS.
  • • Lab tls internal + --ssl-insecure-skip-verify=true → production phải gỡ.

4  SSO & logout flow

sequenceDiagram autonumber participant U as Browser participant A as app A participant B as app B participant OPA as oauth2-proxy A participant OPB as oauth2-proxy B participant Z as zitadel Note over U,Z: SSO — login 1 lần, dùng nhiều app U->>A: GET app-a.authway.lab/ (already logged in) A-->>U: render OK U->>B: GET app-b.authway.lab/ B->>OPB: forwardAuth OPB->>OPB: check cookie .authway.lab → có OPB-->>B: 202 + X-Auth-* headers B-->>U: render OK (không login lại) Note over U,Z: Logout — chấm dứt session ở mọi layer U->>OPA: GET /oauth2/sign_out?rd=https://auth.authway.lab/oidc/v1/end_session OPA->>OPA: clear cookie OPA-->>U: 302 zitadel end_session U->>Z: GET /oidc/v1/end_session Z->>Z: revoke session, audit log Z-->>U: 302 post_logout_redirect
SSO scope

Cookie domain .authway.lab → mọi subdomain cùng parent thấy chung. App khác domain phải login lại (intentional isolation).

Session TTL

oauth2-proxy cookie 168h (default) · Zitadel session policy override → set ngắn (4–8h) cho admin app, dài cho daily tool.

Force logout

Admin disable user trong Zitadel → token introspection fail → oauth2-proxy refresh fail → app force re-login trong < 1 cookie cycle.

9  Migrate app cũ — 2 nhóm pattern

Nhóm A — IP whitelist only 0 LOC

App không có user concept, ai trong mạng thì xài.

  • • Code change: 0 dòng
  • • Chỉ đặt app sau Traefik + oauth2-proxy theo template
  • • IP whitelist GIỮ song song
  • • Tùy chọn: 1 dòng đọc X-Auth-Request-Email để show "Hello {user}"
  • • Thời gian: ~10 phút/app
Nhóm B — Account tự tạo 50–150 LOC

App có DB users, login form, session, data per-user. Quyết định: force re-register, data cũ bỏ.

  • • Bỏ: login/register routes, password column, reset flow, email verify, session tự code
  • • Thêm: middleware trust header, auto-provision user row mới, logout redirect oauth2-proxy
  • • Schema: DROP password_hash, ADD zitadel_subject, TRUNCATE users
  • • Thời gian: 2–4h/app tùy phức tạp

FAQ nhanh

Vì sao chọn Zitadel mà không phải Authentik / Keycloak?↕

Đã debate trong 2 brainstorm trước. Quyết định cuối: user chốt Zitadel vì UI tốt + audit event-sourced + portable OIDC chuẩn. Authentik thắng về DX (forward-auth native) nhưng user ưu tiên Zitadel. Bù lại: dùng oauth2-proxy cho forward-auth → 2 systems, accept trade-off.

Sao mỗi app cần oauth2-proxy riêng? Không dùng 1 instance trung tâm?↕

oauth2-proxy gắn với 1 OIDC client (1 client_id), 1 set redirect URI. Tách per-app: dễ revoke 1 app khi compromise, audit per-app, cookie-secret riêng. Single instance được nếu chấp nhận shared client_id và cookie scope rộng — POC này chọn per-app cho isolation.

Sao không federate Google Workspace mà tạo local user?↕

User decision: local user. Lý do: control hoàn toàn, không phụ thuộc Google, dễ tạo service account, intern không cần Google corp. Trade-off: admin tạo thủ công + maintain passwords. Có thể thêm Google IDP sau làm fallback.

Auth-server down → mọi app khoá?↕

Đúng — single point of failure trong POC. Mitigation: cookie session oauth2-proxy có TTL 168h → user đang login vẫn xài app được trong cookie lifetime, chỉ user mới không login được. Production: HA Zitadel replica + managed Postgres.

Bỏ IP whitelist được không khi đã có Zitadel + MFA?↕

KHÔNG nên. Defense in depth: nếu Zitadel có CVE 0-day, IP whitelist vẫn chặn attacker ngoài subnet. Cost giữ IP whitelist = 0 (đã có firewall rule). Production: thay subnet rule bằng VPN + zero-trust nếu cần WFH.

Scale lên 50 app / 50 dev thì kiến trúc đổi gì?↕

Zitadel + oauth2-proxy xử được dễ. Cần: HA Zitadel (2–3 replica), managed Postgres (Cloud SQL / RDS), Terraform Zitadel provider để tự động tạo Application + grant, Prometheus monitoring oauth2-proxy latency, log shipping ra log central.
