What is Authway

Một "auth brain" duy nhất cho 5 devs — login 1 lần, vào mọi internal tool, có MFA + audit log.

Authway tập trung identity của team vào Zitadel self-host (1 VPS). Mỗi app vibe-code chạy trên VPS riêng có oauth2-proxy làm sole proxy. Dev/admin gõ URL app → oauth2-proxy redirect tới auth.lab.local → login local user + MFA → callback set cookie (per-app, exact host) → oauth2-proxy proxy upstream với header X-Auth-Request-Email. SSO cross-app qua Zitadel session (redirect flash, không re-MFA). Defense in depth: IP whitelist subnet + VPN giữ song song; Zitadel chỉ là layer identity.

🔐 Zitadel OIDC (AGPL self-host) 🪪 MFA enforced (TOTP lab · Passkey prod) 🛡️ IP whitelist + VPN defense in depth 📑 Event-sourced audit log 🚪 1-click offboarding

3  Login pipeline — IAP redirect flow

sequenceDiagram autonumber participant U as Browser participant TA as traefik (app-vps) participant OP as oauth2-proxy (sole proxy) participant TZ as traefik (auth-vps) participant Z as zitadel participant A as app upstream (internal) U->>TA: GET https://nextjs-demo.lab.local/ TA->>TA: entrypoint middleware: strip X-Auth-* (chống spoof) TA->>OP: forward request to oauth2-proxy:4180 OP->>OP: check cookie → none OP-->>U: 302 /oauth2/start?rd=/ U->>OP: GET /oauth2/start OP-->>U: 302 https://auth.lab.local/oauth/v2/authorize?client_id=... U->>TZ: GET /oauth/v2/authorize TZ->>Z: forward Z-->>U: login form U->>Z: username + password Z-->>U: MFA prompt (TOTP cho lab / Passkey cho prod) U->>Z: OTP code / WebAuthn assertion Z-->>U: 302 https://nextjs-demo.lab.local/oauth2/callback?code=... U->>OP: GET /oauth2/callback?code=... OP->>TZ: token exchange (code → tokens) via Basic auth TZ->>Z: forward Z-->>OP: id_token + refresh_token OP->>OP: validate, set cookie (Domain=nextjs-demo.lab.local, exact host) OP-->>U: 302 / (original rd) U->>OP: GET / (with cookie) OP->>OP: cookie valid → proxy upstream + inject X-Auth-Request-* OP->>A: GET / + X-Auth-Request-Email=... A-->>OP: HTML "Hello user@lab.local" OP-->>U: HTML
Highlights
  • • oauth2-proxy là sole proxy, không dùng Traefik forwardAuth (errors-redirect pattern không trigger 302 đúng). oauth2-proxy tự handle login flow + proxy upstream.
  • • Strip X-Auth-* ở entrypoint Traefik (declared trong traefik.yml websecure middlewares) — apply MỌI router, không phụ thuộc label per-app.
  • • Cookie scope per-app (Domain = exact app hostname). Không share cross-app vì public-suffix list chặn `.lab.local`/`.io.vn`.
  • • SSO qua Zitadel session (redirect flash, không re-MFA) — không phải cookie-share. Xem section 4.
  • • MFA enforced trong Zitadel, policy server-side → không bypass được.
  • • Lab: TOTP (self-signed cert block WebAuthn). Production: Passkey + TOTP.
  • • Token chỉ ở oauth2-proxy, app không thấy access_token (--pass-access-token=false).
  • • Lab: --ssl-insecure-skip-verify=true (Zitadel self-signed). Production: gỡ.

4  SSO & logout flow

SSO mechanism

Cookie domain = exact app hostname (per-app, không share). SSO qua Zitadel session — redirect flash, không re-MFA. Áp dụng kể cả app khác parent domain.

Session TTL

oauth2-proxy cookie 168h default + --cookie-refresh=1h → revoke propagate <1h. Zitadel session policy override per-app nếu cần.

Force logout

Admin disable user trong Zitadel → cookie-refresh fail → app force re-login trong <1h. Override ngay: tear-down oauth2-proxy + restart.

9  Migrate app cũ — 2 nhóm pattern

Nhóm A — IP whitelist only 0 LOC

App không có user concept, ai trong mạng thì xài.

  • • Code change: 0 dòng
  • • Chỉ đặt app sau Traefik + oauth2-proxy theo template
  • • IP whitelist GIỮ song song
  • • Tùy chọn: 1 dòng đọc X-Auth-Request-Email để show "Hello {user}"
  • • Thời gian: ~10 phút/app
Nhóm B — Account tự tạo 50–150 LOC

App có DB users, login form, session, data per-user. Quyết định: force re-register, data cũ bỏ.

  • • Bỏ: login/register routes, password column, reset flow, email verify, session tự code
  • • Thêm: middleware trust header, auto-provision user row mới, logout redirect oauth2-proxy
  • • Schema: DROP password_hash, ADD zitadel_subject, TRUNCATE users
  • • Thời gian: 2–4h/app tùy phức tạp

FAQ nhanh

Vì sao chọn Zitadel mà không phải Authentik / Keycloak?↕

Đã debate trong 2 brainstorm trước. Quyết định cuối: user chốt Zitadel vì UI tốt + audit event-sourced + portable OIDC chuẩn. Authentik thắng về DX (forward-auth native) nhưng user ưu tiên Zitadel. Bù lại: dùng oauth2-proxy cho forward-auth → 2 systems, accept trade-off.

Sao mỗi app cần oauth2-proxy riêng? Không dùng 1 instance trung tâm?↕

oauth2-proxy gắn với 1 OIDC client (1 client_id), 1 set redirect URI. Tách per-app: dễ revoke 1 app khi compromise, audit per-app, cookie-secret riêng. Single instance được nếu chấp nhận shared client_id và cookie scope rộng — POC này chọn per-app cho isolation.

Sao không federate Google Workspace mà tạo local user?↕

User decision: local user. Lý do: control hoàn toàn, không phụ thuộc Google, dễ tạo service account, intern không cần Google corp. Trade-off: admin tạo thủ công + maintain passwords. Có thể thêm Google IDP sau làm fallback.

Auth-server down → mọi app khoá?↕

Đúng — single point of failure trong POC. Mitigation: cookie session oauth2-proxy có TTL 168h → user đang login vẫn xài app được trong cookie lifetime, chỉ user mới không login được. Production: HA Zitadel replica + managed Postgres.

Bỏ IP whitelist được không khi đã có Zitadel + MFA?↕

KHÔNG nên. Defense in depth: nếu Zitadel có CVE 0-day, IP whitelist vẫn chặn attacker ngoài subnet. Cost giữ IP whitelist = 0 (đã có firewall rule). Production: thay subnet rule bằng VPN + zero-trust nếu cần WFH.

Scale lên 50 app / 50 dev thì kiến trúc đổi gì?↕

Zitadel + oauth2-proxy xử được dễ. Cần: HA Zitadel (2–3 replica), managed Postgres (Cloud SQL / RDS), Terraform Zitadel provider để tự động tạo Application + grant, Prometheus monitoring oauth2-proxy latency, log shipping ra log central.
