Stack overview

Authway tách thành 2 stack docker-compose độc lập: auth-vps stack (Traefik + Zitadel + Postgres + Mailhog) chạy 1 VPS duy nhất tại 192.168.122.54, và app-vps stack (Traefik + N oauth2-proxy + N app) clone deploy trên mỗi VPS app khác (lab: 192.168.122.55). Header spoof chống ở entrypoint-level middleware Traefik (strip X-Auth-*) — apply mọi router, không cần lặp per-app. oauth2-proxy đóng vai sole proxy cho từng app: tự handle login flow + cookie + proxy upstream nội bộ.

auth-vps: 4 service app-vps: 5 service (cho 2 demo app) Network: bridge per stack TLS: self-signed (lab) · ACME (prod)

①traefik edge · auth-vps

Edge proxy cho auth.lab.local — TLS termination, IP whitelist middleware, routing tới Zitadel

image: traefik:v3.4 ports: 80, 443

⚡ Tác dụng — vì sao tồn tại

Là cửa duy nhất internet/intranet chạm Zitadel. Mọi request HTTPS → terminate TLS ở đây → forward h2c tới Zitadel. Nếu thiếu Traefik: Zitadel phải tự handle TLS (phức tạp cert renewal), không lọc được IP, không strip header spoof từ client.

Giá trị 1 · Single edge
Tất cả traffic tới auth.lab.local đi qua 1 chỗ → 1 nơi để log, rate-limit, audit. Không có Traefik = Zitadel expose trực tiếp internet.
Giá trị 2 · Defense layer
IP whitelist + security-headers + (sau này) rate-limit. Tách concern: Zitadel lo identity, Traefik lo network/edge security.
Giá trị 3 · TLS automation
ACME auto-renew Let's Encrypt khi prod. Cert resolver internal cho lab tự ký. App không cần biết về cert.

②zitadel OIDC issuer · MFA · audit

Core identity provider — local users, login policy MFA enforced, OIDC + OAuth2.1 endpoints, event-sourced audit

image: ghcr.io/zitadel/zitadel:v4.15.3 internal port: 8080

⚡ Tác dụng — vì sao tồn tại

Trái tim hệ thống — nơi duy nhất biết "user là ai, đăng nhập được không, có MFA chưa". Toàn bộ apps không tự lưu password — đẩy hết cho Zitadel. Nếu thiếu Zitadel: mỗi app tự build login/MFA/passkey/audit → vibe-coder không kham nổi, lockdown rời rạc, không SSO.

Giá trị 1 · Single source of truth
1 user record dùng cho tất cả apps. Disable user 1 lần → mất quyền mọi nơi trong <1h (cookie refresh). Không có Zitadel = N database user phải đồng bộ tay.
Giá trị 2 · MFA + Passkey grade
Passkey/TOTP enforce policy-level. App downstream không cần biết MFA tồn tại — Zitadel chặn ở login. Loại bỏ password-only attack surface toàn cụm.
Giá trị 3 · Audit trail bất biến
Event-sourced → mọi login, mọi grant, mọi password reset đều có dấu vết append-only. Forensic được "ai làm gì lúc nào" mà không cần build log system riêng.

③postgres persistence · audit · sessions

Persistence cho Zitadel: users, projects, applications, grants, sessions, audit events

image: postgres:16-alpine port: 5432 (local only)

⚡ Tác dụng — vì sao tồn tại

Bộ nhớ persistent cho Zitadel. Zitadel stateless — restart không mất gì vì state nằm hết ở đây: users, hashed password, MFA enrollment, project/app config, OIDC client secret, mọi audit event. Mất Postgres = mất TOÀN BỘ identity store.

Giá trị 1 · Single durable store
1 dump = backup nguyên hệ thống auth (không có "config nằm rải rác file YAML"). Restore 1 lệnh = khôi phục full state.
Giá trị 2 · Event store append-only
Audit events bất biến — không UPDATE/DELETE. Replay event = tái dựng state bất kỳ thời điểm (forensic, debug "ai đã sửa policy hôm 15/3").
Giá trị 3 · Critical SPOF · cần plan HA
Đây là điểm rủi ro lớn nhất. POC chấp nhận single-node. Production: managed Postgres (Cloud SQL/RDS) với PITR + replica + automated backup.

④traefik edge · app-vps

Mỗi app-vps có 1 Traefik riêng. Routes thẳng host → oauth2-proxy:4180 (oauth2-proxy là sole proxy, tự handle auth + upstream)

image: traefik:v3.4 ports: 80, 443

⚡ Tác dụng — vì sao tồn tại (khác Traefik auth-vps)

Edge cho mỗi VPS app + nơi chặn header spoof. Khác với Traefik auth-vps: nó không route tới app trực tiếp — luôn forward sang oauth2-proxy. Nếu thiếu Traefik app-vps: app phải tự expose 443, tự strip header, tự cert. Vibe-coder không làm nổi đồng đều.

Giá trị 1 · Strip X-Auth-* incoming
Middleware strip-auth-in ở entrypoint level → mọi request từ client BỊ XOÁ header X-Auth-* trước khi tới oauth2-proxy. Chống attacker giả header để bypass auth.
Giá trị 2 · TLS + routing đồng nhất
1 template Traefik cho mọi app-vps — vibe-coder copy compose, đổi env, app live. Không phải nghĩ về cert, redirect, CSP.
Giá trị 3 · Tách concern
Traefik lo "đưa request vào đúng oauth2-proxy"; oauth2-proxy lo "user đã auth chưa"; app lo "logic nghiệp vụ". 3 lớp rõ ràng, fail-mode khoanh vùng được.

Routing pattern (oauth2-proxy = sole proxy)

Entrypoint middleware
strip-auth-in + security-headers apply MỌI router → chống spoof không phụ thuộc per-router config
Router rule
Host(APP_HOSTNAME) → service oauth2-proxy-<app>:4180
Upstream
oauth2-proxy có --upstream=http://<app>:<port>, internal docker network
App container
KHÔNG có Traefik label — chỉ reachable qua oauth2-proxy internal
  • Pattern đổi: POC ban đầu thử forwardAuth + errors-redirect (Traefik route app trực tiếp), nhưng errors middleware không trigger 302 đúng. Đổi sang oauth2-proxy làm sole proxy.
  • oauth2-proxy tự xử login flow: chưa auth → 302 /oauth2/start → Zitadel → callback → set cookie → proxy upstream với X-Auth-* headers.
  • Strip middleware ở entrypoint level (declared trong traefik.yml) — apply mọi router, không cần lặp lại per-app.

⑤oauth2-proxy OIDC client · cookie · header inject

Sole proxy trên mỗi app-vps. Mỗi app = 1 instance riêng (1 OIDC client + 1 cookie secret). KHÔNG share cookie cross-app (public-suffix safe).

image: quay.io/oauth2-proxy/oauth2-proxy:v7.7.1 internal port: 4180

⚡ Tác dụng — vì sao tồn tại (component LÕI của IAP pattern)

"OIDC translator" — biến app zero-auth thành app có auth mà không sửa code. Tự handle login flow, cookie, refresh token, inject header. App downstream chỉ đọc X-Auth-Request-Email = 1 dòng code. Nếu thiếu oauth2-proxy: mỗi app phải tự code OIDC client (PKCE, callback, cookie, refresh, MFA flow) — static HTML thì BẤT KHẢ THI.

Giá trị 1 · App không biết OIDC tồn tại
Static HTML, Streamlit, FastAPI demo — không cần SDK Zitadel, không cần lib OIDC. Vibe-coder ship app mới <15 phút.
Giá trị 2 · Per-app isolation
Mỗi app có instance riêng = OIDC client riêng = cookie domain exact host riêng. App A bị XSS không leo sang app B (cookie không share). Revoke 1 app = 1 client_secret, không ảnh hưởng app khác.
Giá trị 3 · Token isolation
--pass-access-token=false → app KHÔNG cầm access_token. App bị XSS không exfil token được. --cookie-refresh=1h → disable user trong Zitadel mất quyền <1h tự động.

Vai trò

OIDC client
--provider=oidc + issuer URL Zitadel, Auth Method Basic (client_secret) + Refresh Token
Cookie (per-app)
cookie-domain = exact app hostname (không phải parent — public suffix có thể chặn). Mỗi app cookie độc lập.
SSO mechanism
SSO qua Zitadel session (redirect flash, no MFA re-prompt), KHÔNG qua cookie share.
Header inject
--set-xauthrequest=true → X-Auth-Request-Email + User + Preferred-Username vào upstream
  • --pass-access-token=false — app KHÔNG nhận access_token (giảm XSS exfil risk)
  • --cookie-refresh=1h — revoke user trong Zitadel propagate <1h (red-team #10)
  • --whitelist-domain=auth.lab.local — cho phép logout rd chain về Zitadel end_session
  • --ssl-insecure-skip-verify=true CHỈ cho lab (Zitadel self-signed). Production: gỡ.
  • --skip-provider-button=true — không hiện màn hình chọn provider, redirect thẳng Zitadel
  • extra_hosts trong compose map auth.lab.local → IP auth VPS (container DNS không thấy /etc/hosts host)
  • Cookie secret: openssl rand -base64 32 | tr -d '/+=' | cut -c1-32 (đúng 32 chars) — generate riêng mỗi app

⑥app downstream vibe-code · đọc header

Bất kỳ app gì: Next.js / static HTML / Streamlit / Python. Code change ≤ 5 LOC để đọc X-Auth-Request-Email

image: tùy app port: tùy

⚡ Tác dụng — vì sao tồn tại (consumer của toàn bộ stack)

Đích cuối của mọi request — nơi business logic chạy. Hợp đồng cực đơn giản: "Tin header X-Auth-Request-Email, đừng tự làm auth". App không lưu password, không có trang login, không quản session. Stack auth toàn bộ là "vô hình" với app code — đây chính là giá trị KISS của IAP pattern.

Giá trị 1 · Migration cost cực thấp
Nhóm A (IP whitelist only): 0 LOC. Nhóm B (app tự auth): bỏ login/password + middleware đọc header = 50-150 LOC. So với tự tích OIDC SDK: 500+ LOC + maintain.
Giá trị 2 · Polyglot friendly
Next.js, FastAPI, Streamlit, static HTML, Grafana, pgAdmin — tất cả đều "đọc header" được. Không bị khoá stack/ngôn ngữ. Vibe-code bằng tool nào cũng dán vào được.
Giá trị 3 · FK ổn định qua subject ID
Lưu zitadel_subject (UUID stable) làm FK thay vì email. User đổi email = data không vỡ. Đây là lý do bắt buộc đọc X-Auth-Request-User chứ không chỉ email.

Header contract

X-Auth-Request-Email
Email của user — chính
X-Auth-Request-User
Zitadel subject ID (UUID) — stable, dùng làm FK
X-Auth-Request-Preferred-Username
Username friendly
(KHÔNG)
Access token — đã --pass-access-token=false
  • Trust contract: app TIN tưởng header này vì Traefik đã strip incoming + iap-auth chỉ set khi cookie valid
  • Code Next.js: const email = headers().get('x-auth-request-email')
  • Code Express: app.use((req,_,next) => { req.user = { email: req.headers['x-auth-request-email'] }; next() })
  • Code Python FastAPI: dependency đọc request.headers.get('x-auth-request-email')

⑦backup cron daily · 30-day retention

Postgres dump daily 02:00 → /opt/authway/backup/zitadel-YYYYMMDD-HHMM.dump

⚡ Tác dụng — vì sao tồn tại

Bảo hiểm phá sản. Postgres = SPOF chứa toàn bộ users + audit + config. Mất disk = mất hết identity → mọi app cluster mất quyền truy cập. Backup daily + restore test = phòng tuyến cuối khi disk fail, ransomware, hoặc operator xoá nhầm.

Giá trị 1 · RPO 24h chấp nhận được
Internal tool, <1k user — mất 1 ngày event log OK. Production critical thì cần WAL streaming (RPO <1min) — nâng cấp khi cần.
Giá trị 2 · Restore test = backup thật
Backup không restore được = không có backup. Monthly restore test vào staging → verify integrity + đo restore time (RTO baseline).
Giá trị 3 · Offsite chống ransomware
Production phải ship sang NAS / S3 / MinIO khác VPS. Local backup cùng disk = vô nghĩa khi disk hỏng cả 2 file.

Lưu ý

  • Format custom (binary) — restore qua pg_restore, không phải psql
  • Retention 30 ngày local; production cần offsite (rsync NAS / MinIO / S3)
  • Restore test monthly vào staging container — verify integrity
  • Log đường /var/log/authway-backup.log — alert nếu missed run

⑧systemd unit auto-restart on boot

Đảm bảo stack tự up sau reboot VPS

⚡ Tác dụng — vì sao tồn tại

Tự phục hồi sau reboot. VPS kernel update / power loss / migrate host → docker daemon up nhưng compose stack KHÔNG tự up. Không có systemd unit = sau reboot, admin phải SSH chạy docker compose up tay → downtime kéo dài + dễ quên.

Giá trị 1 · Zero-touch boot
Reboot xong, Zitadel up automatic trong 30-60s. Không cần on-call ngồi chờ SSH.
Giá trị 2 · Standard ops interface
systemctl status/start/stop authway — interface chuẩn ops biết. Không cần training thêm "compose dir nằm đâu".
Giá trị 3 · Dependency ordering
After=docker.service network-online.target → đảm bảo network + docker ready trước khi start. Tránh race condition khiến container không bind port được.

Unit file

[Unit]
Description=Authway Zitadel stack
After=docker.service network-online.target
Requires=docker.service

[Service]
Type=oneshot
RemainAfterExit=yes
WorkingDirectory=/opt/authway/infra/auth-vps
ExecStart=/usr/bin/docker compose up -d
ExecStop=/usr/bin/docker compose down
User=authops

[Install]
WantedBy=multi-user.target
  • oneshot + RemainAfterExit — Docker tự manage container lifecycle, systemd chỉ "khởi tạo lúc boot"
  • Verify sau reboot: systemctl status authway + docker compose ps
  • App-vps tương tự — mỗi VPS có 1 unit riêng trỏ vào compose dir tương ứng
